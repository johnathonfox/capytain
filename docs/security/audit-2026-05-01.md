# Security Audit Report - 2026-05-01

## Executive Summary

This audit identified 5 security issues across the qsl codebase, including 2 high-severity vulnerabilities that should be addressed before release. The most critical issues are missing token zeroization on drop and lack of token revocation on logout, both of which could allow attackers to gain persistent access to user accounts. Additionally, the application is missing a Content Security Policy (CSP) which leaves it vulnerable to XSS attacks. The codebase shows good security practices in OAuth2 implementation and token handling, but these gaps need to be addressed for production readiness.

## Detailed Findings

### QSL-SEC-001: Missing Token Zeroization (High)

**Context:** The `AccessToken` and `RefreshToken` types in `crates/auth/src/tokens.rs` are not zeroized on drop, which could lead to tokens being leaked through memory dumps or crash reports.

**Threat Model:** If tokens are leaked through logs, memory dumps, or crash reports, attackers could gain access to user accounts. The `AccessToken` and `RefreshToken` types are not zeroized on drop.

**Suggested Fix:** Add `Drop` implementations that zeroize the token contents using the `zeroize` crate. The `AccessToken` and `RefreshToken` structs should implement `Drop` to clear their inner strings.

### QSL-SEC-002: Missing Token Revocation on Logout (High)

**Context:** When users remove accounts, refresh tokens are deleted from the keychain but not revoked at the OAuth provider in `apps/desktop/src-tauri/src/commands/accounts.rs`.

**Threat Model:** When users remove accounts, refresh tokens are deleted from the keychain but not revoked at the OAuth provider. This leaves the tokens potentially usable by attackers who gain access to the keychain.

**Suggested Fix:** Add token revocation to the `accounts_remove` command. Before deleting the refresh token from the keychain, make a call to the provider's token revocation endpoint (e.g., Google's `https://oauth2.googleapis.com/revoke`).

### QSL-SEC-003: State Parameter Validation (Medium)

**Context:** The state parameter validation in `crates/auth/src/flow.rs` correctly checks for mismatches but could provide more explicit error handling.

**Threat Model:** The state parameter is validated against the expected value, but the validation is only done after the redirect is received. While the code correctly checks for mismatch, it doesn't provide a clear indication of what happens if the state is invalid.

**Suggested Fix:** The code should be more explicit about what happens when state validation fails. Consider adding a more specific error type or logging more details about the mismatch.

### QSL-SEC-004: Missing Content Security Policy (Medium)

**Context:** The Tauri configuration in `apps/desktop/src-tauri/tauri.conf.json` has `csp: null` which means no Content Security Policy is set.

**Threat Model:** The Tauri configuration has `csp: null` which means no Content Security Policy is set. This leaves the application vulnerable to XSS attacks.

**Suggested Fix:** Set a proper CSP in `tauri.conf.json` with `default-src 'self'` and appropriate script-src directives. The CSP should be set to prevent XSS and other injection attacks.

### QSL-SEC-005: Input Validation (Medium)

**Context:** The `AccountsAddOauthInput` struct in `apps/desktop/src-tauri/src/commands/accounts.rs` accepts user-provided provider and email parameters without validation.

**Threat Model:** The `AccountsAddOauthInput` struct accepts user-provided provider and email parameters without validation. If these are not properly sanitized, they could be used in injection attacks or to bypass access controls.

**Suggested Fix:** Add validation to ensure the provider is one of the known providers and that the email address is properly formatted.

## Deferred Findings

The following findings are not blocking for 0.1 but should be addressed in future releases:

1. **Unmaintained Dependencies**: 19 unmaintained crates identified, primarily GTK3 bindings. These are likely not directly used in core functionality but could be a concern for long-term maintenance.

2. **Token Scope Minimization**: Gmail uses the full `https://mail.google.com/` scope which provides full access to the user's email. While required for current implementation, consider using narrower scopes if possible.

3. **Build-time Secrets Handling**: The build script in `crates/auth/build.rs` loads secrets from environment variables and `.env` files. Ensure `.env` files are properly gitignored to prevent accidental commits.

## Summary

The qsl codebase implements OAuth2 flows with PKCE correctly and follows security best practices for token storage. However, there are several critical issues that need to be addressed:

1. **Missing token zeroization** - Tokens should be zeroized on drop to prevent memory leaks
2. **Missing token revocation on logout** - Refresh tokens should be revoked at the provider when accounts are removed
3. **Missing CSP** - The Tauri configuration lacks a Content Security Policy, making the application vulnerable to XSS

The codebase shows good attention to security in many areas, particularly around OAuth2 implementation and token handling, but these gaps need to be addressed before production deployment.
